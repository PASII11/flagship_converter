# Дизайн: публикация Flagship Converter на GitHub

**Дата:** 2026-07-19
**Статус:** одобрен

## Цель

Оформить публичный репозиторий `github.com/PASII11/flagship_converter` так, чтобы
пользователь мог понять, что это за программа, скачать её со страницы Releases
(установщик или portable-ZIP) и узнавать о новых версиях из самого приложения.

## Контекст

- Репозиторий уже публичный, код на main запушен; README, LICENSE и релизов нет.
- Сборка PyInstaller (`build.py`) — onedir, ~1.3 ГБ: внутри ffmpeg, wkhtmltopdf,
  torch, модели docling. Лимит GitHub — 2 ГБ на файл релиза, сжатые артефакты
  укладываются.
- Принятые решения: дистрибутив — установщик **и** portable-ZIP; README на
  английском; лицензия MIT; сборка релизов локально скриптом (CI — потом);
  сейчас — проверка обновлений с кнопкой скачивания, полное автообновление — потом.

## 1. README и оформление репозитория

`README.md` (английский):

- Название + слоган: privacy-first local file converter for Windows — everything
  runs on your machine, nothing leaves it.
- Скриншот приложения (`docs/assets/screenshot.png`) — снимаем при реализации.
- Возможности (по фактическим конвертерам в `core/converters/`): изображения
  (включая HEIC/AVIF), видео (пережатие до целевого размера файла, извлечение
  аудио, GIF), аудио, документы (PDF ↔ DOCX, Markdown/HTML → PDF, OCR через
  docling/rapidocr).
- Download: ссылка на Releases, что выбрать (Setup vs portable ZIP), системные
  требования (Windows 10/11 x64, ~3 ГБ диска).
- Build from source: uv sync, установленные ffmpeg и wkhtmltopdf,
  `uv run python build.py`.
- Секция License.

Описание и topics в шапке GitHub (About) заполняются вручную владельцем либо
через `gh repo edit` после установки gh CLI.

Гигиена репозитория: `build_tools/docling_models/` (скачанные модели, >1 ГБ)
добавляется в `.gitignore` — в git они попадать не должны.

## 2. Лицензия и версионирование

- `LICENSE` — MIT, copyright владельца репозитория.
- Единый источник версии — новый `src/flagship_converter/version.py`
  (`__version__: str`). `pyproject.toml` держим синхронным; релизный скрипт
  проверяет соответствие.
- Первый публичный релиз — **v1.0.0** (bump с 0.1.0). Каждый релиз = git-тег
  `vX.Y.Z` + GitHub Release.

## 3. Установщик и portable-ZIP

- `packaging/installer.iss` (Inno Setup): вход — готовая `dist/FlagshipConverter`,
  выход — `FlagshipConverter-Setup-<версия>.exe` (LZMA2-сжатие, ~700 МБ).
- Ярлыки в меню «Пуск» + опционально на рабочем столе, деинсталлятор.
- **Фиксированный AppId (GUID)** — новая версия ставится поверх старой; это
  фундамент будущего полного автообновления.
- Portable: `dist/FlagshipConverter`, запакованный в
  `FlagshipConverter-<версия>-portable-win64.zip`.
- Требования на машине сборки: Inno Setup (ISCC.exe в PATH или стандартном
  месте), gh CLI. Скрипт релиза проверяет наличие обоих и падает с внятной
  инструкцией по установке.

## 4. Скрипт релиза

`release.py` (запуск: `uv run python release.py`):

1. Проверяет: рабочее дерево чистое; версия в `version.py` == `pyproject.toml`;
   тега `v<версия>` ещё нет на remote.
2. Собирает приложение (`build.py`); флаг `--skip-build` использует готовый dist.
3. Пакует ZIP, компилирует установщик через ISCC.
4. Считает SHA256 обоих артефактов → `SHA256SUMS.txt`.
5. Создаёт тег, пушит его и публикует релиз одной командой
   `gh release create v<версия> <артефакты> --notes ...`.

Любая ошибка на любом шаге — остановка с понятным сообщением; никаких
полуопубликованных релизов (release создаётся последним шагом, когда все
артефакты готовы).

## 5. Проверка обновлений в приложении

Новый модуль `src/flagship_converter/core/update_checker.py`:

- При старте приложения фоновый поток (QThread) делает один GET
  `https://api.github.com/repos/PASII11/flagship_converter/releases/latest`
  с таймаутом ~5 с. Только stdlib (`urllib.request`) — новых runtime-зависимостей
  не добавляем.
- Сравнение версий: разбор `tag_name` вида `vX.Y.Z` в кортеж чисел, сравнение
  с локальной `__version__`. Некорректный тег — считаем, что обновления нет.
- Если версия новее — в главном окне появляется ненавязчивая кнопка/баннер
  «Update available: vX.Y.Z»; клик открывает страницу релиза в браузере
  (`QDesktopServices.openUrl`).
- Ошибки сети, таймаут, rate-limit 403, битый JSON — молча пропускаем; проверка
  никогда не блокирует UI и не роняет приложение.
- Приватность: единственный анонимный запрос к GitHub API, ничего не
  отправляется; упомянуть в README.

## Тестирование

- pytest: сравнение версий (граничные случаи: равные, prerelease-подобный
  мусор в теге, некорректный формат), разбор ответа API на моках, «ошибка сети
  → нет обновления, нет исключения».
- `release.py --dry-run`: проходит проверки и печатает план без побочных
  эффектов.
- Ручная проверка: установка через Setup.exe, запуск, работа кнопки обновления
  при подложенной старой локальной версии.

## За рамками итерации (будущее)

- Полное автообновление: приложение само качает Setup.exe и запускает
  `/SILENT`-установку поверх (AppId уже это позволяет).
- Сборка и публикация релизов через GitHub Actions по пушу тега.
